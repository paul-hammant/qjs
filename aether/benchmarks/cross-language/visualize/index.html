How to read this table
Throughput — Messages processed per second (higher is better). Median of multiple runs.
Latency — Nanoseconds per message (lower is better). Inverse of throughput.
Memory — Peak resident set size in megabytes. Includes runtime overhead (JVM ≈ 50–100 MB, BEAM ≈ 60 MB, native ≈ 1–5 MB).
Relative — Percentage of the fastest language's throughput. 100% = fastest, 50% = half as fast. The bar visualizes this.
CV% — Coefficient of variation (standard deviation ÷ mean × 100). Measures run-to-run stability: ●<5% stable, ●5–15% moderate variance, ●>15% noisy (thermal throttling, OS scheduling).
Range — Minimum and maximum throughput observed across all runs. Wider range = less predictable.
Efficiency — Throughput divided by memory (M msg/s per MB). Higher means more work done per unit of memory. Favors lightweight runtimes.
Based on the Savina Actor Benchmark Suite (Imam & Sarkar, Rice University, 2014).
All languages compiled with highest optimization flags. Median of N runs reported. JVM/BEAM languages receive warmup runs before timed measurement.
Throughput = operations / wall-clock elapsed (CLOCK_MONOTONIC). Memory = peak RSS (/usr/bin/time). CV% = coefficient of variation (σ/μ).
